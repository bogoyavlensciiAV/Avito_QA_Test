# Отчет о найденных багах (Avito API)

По результатам автоматизированного тестирования (выполнено 20 тест-кейсов) обнаружены следующие отклонения от ожидаемого поведения. Ниже в формате сводной матрицы представлена таблица найденных багов (отсортирована по критичности).

| # | Название бага | Severity | Priority | Описание | Шаги воспроизведения | Фактический результат | Ожидаемый результат |
|---|---|---|---|---|---|---|---|
| 1 | **Возможность создать объявление с отрицательной ценой** | Blocker | Highest | Сервис позволяет создать объявление с `price: -100` (и любым отрицательным значением), если в объекте `statistics` все поля > 0. Это нарушает бизнес-логику и может привести к мошенничеству/некорректным объявлениям. | 1. `POST /api/1/item`<br>2. Тело:<br>`{`<br>`"sellerID": 222222,`<br>`"name": "Товар",`<br>`"price": -100,`<br>`"statistics": { `<br>&nbsp;&nbsp;`"likes": 1,`<br>&nbsp;&nbsp;`"viewCount": 1,`<br>&nbsp;&nbsp;`"contacts": 1`<br>`}`<br>`}` | 200 OK<br>`{"status": "Сохранили объявление - 70...cf0"}`<br><br>Объявление создаётся с ценой -100. При последующем GET-запросе цена вернётся как -100. | 400 Bad Request с сообщением типа "цена не может быть отрицательной". Объявление не должно сохраняться. |
| 2 | **Искажение данных (статистики) при создании объявления** | Blocker | Highest | Сохраненные данные не совпадают с отправленными при создании. Статистика (`likes`, `viewCount`, `contacts`) обнуляется или полностью перезаписывается случайными значениями (например, всегда отдаётся `{contacts: 3, likes: 1, viewCount: 2}`) на стороне сервера, что полностью ломает функционал аналитики. | 1. Отправить `POST /api/1/item` с телом, где метрики явно отличны от 1 или 0 (например, `likes: 50`, `viewCount: 15`).<br>2. Вызвать `GET /api/1/item/{id}` или `GET /api/1/statistic/{id}`. | 200 OK<br><br>Возвращаются искаженные данные статистики, переопределенные сервером (например, `likes: 1`, `viewCount: 2`), отличные от отправленных в теле POST. | Все отправленные поля должны корректно сохраняться в БД и возвращаться без искажений. |
| 3 | **Запись нулей трактуется системой некорректно (обнуление)** | Major | High | Создание объявления с нулевыми значениями (`price: 0`, или метрики статистики: 0) приводит к тому, что при последующем запросе эти счетчики не совпадают с возвращаемыми (система их заменяет на левые числа). То есть, объявление не может стоить "Бесплатно" без бага. | 1. `POST /api/1/item` с `price: 0` или статистикой в нулях.<br>2. Вызвать `GET /api/1/item/{id}`. | 200 OK<br><br>Возвращенные значения не равны нулю (затираются мусорными значениями бэкенда, например "3"). | 200 OK<br><br>Поля с нулями должны корректно приниматься, сохраняться в БД и отдаваться при GET как `0`. |
| 4 | **Несоответствие схемы ответа при создании объявления (отсутствие JSON)** | Major | High | Согласно документации (Postman), при успешном создании объявления API должно возвращать JSON-объект самого созданного объявления (`{"id": "...", "sellerId": ...}`). По факту API возвращает неструктурированный текстовый фрейм внутри объекта. | 1. Вызвать `POST /api/1/item`<br>2. Передать любое валидное тело JSON объявления. | 200 OK<br><br>`{"status": "Сохранили объявление - <uuid>"}` | 200 OK<br><br>Возвращается полная JSON-схема `Item` с назначенным уникальным "id" в качестве ключа. |
| 5 | **Получение объявления по уникальному ID возвращает массив** | Normal | Medium | Поиск конкретного объявления по его уникальному `id` возвращает массив JSON, состоящий из 1 объекта, хотя ожидается строго 1 JSON-объект. Это противоречит REST-архитектуре поиска объекта по ключу ресурса (Postman так же копирует этот баг в доках). | 1. Узнать валидный `<id>`.<br>2. Вызвать `GET /api/1/item/{id}`. | 200 OK<br><br>Ответ представляет собой JSON-массив:<br>`[ { "id": "...", "name": ... } ]` | 200 OK<br><br>Ответ должен быть единичным JSON-объектом: `{ "id": "...", "name": ... }` |
| 6 | **Система осуществляет поиск по отрицательному `sellerID`** | Normal | Medium | Эндпоинт позволяет без ошибки передавать и производить запрос по отрицательным значениям параметра `sellerID` (например, `-111111`) без выдачи предупреждения о неверном формате параметра пути. | 1. Вызвать `GET /api/1/-111111/item`. | 200 OK<br><br>Возвращается пустой список `[]` (либо результаты, если такие багованные объявления существуют). | 400 Bad Request с сообщением об ошибке валидации формата `sellerID`. |
| 7 | **Несоответствие структурных типов данных в документации Postman (POST)** | Minor | Low | В коллекции Postman (вкладка `Body -> raw` для `POST /api/1/item`) пример полезной нагрузки (JSON) указан с оберткой переменных типа `Integer` в строку-кавычки, например: `"sellerID": "<integer>"`, `"price": "<integer>"`. Строгая передача дословного JSON с оберткой выдаст ошибку. | 1. Взять raw-body прямо из Postman коллекции.<br>2. Передать числа как строки:<br>`"sellerID": "111111"`.<br>3. `POST /api/1/item`. | 400 Bad Request<br><br>`{"status": "не передано тело объявления"}` | 200 OK и корректно созданное объявление (либо официальное исправление документации Postman, где следует убрать строки-кавычки вокруг интов). |

*Комментарий от автоматизации: В связи с блокером при создании сущностей (Баг #4 — возвращается статус-строка, а не JSON), пайплайн автотестов был усилен динамическим экстрактором паттернов, который вытаскивает UUID "на лету" (метод `extractId()`), поэтому проверки GET-операций (баги 2, 3 и 5) успешно доходят до конца даже в условиях нерабочей выдачи.*
